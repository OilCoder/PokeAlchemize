### Learnings

**FLUX Kontext vs LoRA — trade-off real:**
- Kontext garantiza identidad (parte del sprite → anatomía siempre correcta) pero es conservador: algunos Pokémon solo se recolorean incluso a guidance=7.0
- LoRA tiene libertad artística real pero pierde identidad en Pokémon menos conocidos (bulbasaur_fire generó una tortuga genérica, pikachu_ghost un ratón genérico)
- Mejores resultados Kontext: `gengar_ice` (glacial translúcido perfecto), `charmander_water` (aletas + piel húmeda)
- Mejores resultados LoRA: `gengar_ice`, `squirtle_electric`

**InstantX FLUX IP-Adapter — por qué falla en diffusers 0.37.1:**
1. `load_ip_adapter` verifica orden exacto de keys: `["image_proj", "ip_adapter"]`. El archivo `ip-adapter.bin` las tiene en orden inverso → `ValueError`. Fix aplicado: reordenar keys en el archivo de caché con `torch.save`.
2. Tras fix #1: diffusers hardcodea `CLIPVisionModelWithProjection.from_pretrained()` como image encoder para FLUX IP-Adapter. InstantX usa SigLIP (729 tokens de posición), CLIP espera 730 → `RuntimeError: size mismatch`. Sin fix limpio.
3. Raíz del problema: diffusers 0.37.1 tiene soporte FLUX IP-Adapter incompleto — no maneja encoders SigLIP en ese code path.

**Alternativa viable no explorada:** `XLabs-AI/flux-ip-adapter` usa CLIP como encoder → compatible con el loader de diffusers. Calidad de anclaje de identidad posiblemente menor que SigLIP pero funcional.

**Velocidad de inferencia a 1024px (Kontext v8, RTX 4080):**
- Primer imagen: ~4 min (compilación CUDA kernels en primer run)
- Imágenes siguientes: ~1:15 por imagen a 30 steps (~2.5s/step en régimen)
- Total 5 imágenes: ~9 min tras warmup

**WSL2 RAM:** expandido a 30GB via `.wslconfig` (`memory=30GB, swap=8GB`) para evitar OOM en cargas grandes.

## 23:40 — Kontext guidance extremo + IP-Adapter v2 abandonado + decisión de cambio a ControlNet

### Technical changes
- `debug/dbg_flux_ipadapter_v2.py` — XLabs-AI/flux-ip-adapter + WiroAI LoRA + standalone CLIP encoder. Requirió 6 workarounds acumulados (ver Learnings). Generó imágenes pero perdía forma del Pokémon. Abandonado.
- `debug/dbg_flux_kontext_v9.py` — guidance_scale 7.0→10.0. Mejora visual notable en bulbasaur y pikachu vs v8.
- `debug/dbg_flux_kontext_v10.py` — guidance_scale 10.0→30.0 (extremo). Test con bulbasaur+pikachu primero, luego 5 completos. Confirmó límite arquitectural de Kontext.

### Design decisions
- **IP-Adapter v2 (XLabs) abandonado**: CLIP-L captura contenido semántico, no identidad estructural fina. Los Pokémon perdían completamente la forma aunque la identidad semántica estuviera presente. Sin workaround posible — es una limitación del encoder.
- **Kontext guidance=30.0 descartado como solución**: el modelo solo cambia color/textura para rasgos con geometría incompatible (bulbo de bulbasaur → volcán falla siempre). Funciona solo cuando la transformación mapea a la geometría existente (llama de charmander → aleta, picos de gengar → carámbanos).
- **Nueva dirección: SDXL + ControlNet Lineart**: ControlNet Lineart extrae líneas del sprite y las usa como guía estructural suave. Con peso bajo (0.3–0.5) permite cambios anatómicos; con peso alto (0.8+) congela estructura como Kontext. Punto de control ajustable que no tenemos con ningún otro approach probado.

### Learnings

**Límite arquitectural de Kontext confirmado experimentalmente:**
- Kontext condiciona sobre el latente completo de la imagen de entrada. La geometría queda fijada antes de que el texto influya.
- Guidance_scale solo amplifica la influencia del texto sobre color/textura/detalles superficiales — no puede reemplazar una forma por otra.
- Casos donde Kontext funciona para anatomía: cuando el prompt describe transformar un rasgo que tiene una forma "conocida" como reemplazable (llama → aleta, pico → carámbano).
- Casos donde falla: geometría "neutra" como un bulbo, un caparazón, mejillas redondas — el modelo no tiene referencia de cómo transformarlos en otra geometría.

**IP-Adapter v2 — workarounds acumulados antes del abandono:**
1. `torch.no_grad()` alrededor de `load_ip_adapter` — evita error de gradient en tensores nf4
2. CLIP encoder standalone fuera del pipeline — evita device mismatch por hooks de cpu_offload
3. Pre-encodear negative embeds — evita que el pipeline llame a `pipe.image_encoder` (device mismatch)
4. `.unsqueeze(1)` en embeds — `encoder_hid_proj` espera `[batch, num_images, embed_dim]` (3D)
5. `gc.collect()` + `del embeds` entre imágenes — evita OOM acumulativo en VRAM
6. Separar carga del pipeline en dos fases — evitar conflictos de offload

**CLIP-L vs identidad estructural:** CLIP-L encodea significado semántico ("esto es un ratón amarillo con orejas"). No encodea geometría fina (posición exacta de las mejillas, forma de la cola, proporciones). El IP-Adapter usa ese embedding para guiar la generación → resultado semánticamente correcto pero estructuralmente genérico.

**Velocidad Kontext v10 (guidance=30.0, RTX 4080):**
- ~3:30 por imagen (igual que v9 — guidance_scale no afecta velocidad de inferencia)
- 5 imágenes completas: ~18 min total

## 22:20 — Configuración final SDXL+ControlNet validada + arquitectura de 3 especialistas

### Design decisions
- **Config final de imagen validada** — IMAGE_SIZE=768 (resolución de entrenamiento del LoRA), STEPS=50, GUIDANCE_SCALE=8.0, LORA_SCALE=0.6, CONTROLNET_SCALE=0.55, EulerDiscreteScheduler+Karras sigmas
- **Lineart B seleccionado** — Canny nativo a 475px (resolución sprite) + LANCZOS upscale a 768px. Líneas limpias sin doble borde, suavizadas como anime lineart
- **prompt/prompt_2 diferenciados** — prompt corto para CLIP-L (estilo + tipo), prompt_2 en lenguaje natural para OpenCLIP-G (descripción detallada de la transformación). Motivo: LoRA entrenado con captions BLIP, no con tags booru
- **negative/negative_2 diferenciados** — negative corto para CLIP-L (colores/tipos a suprimir), negative_2 descriptivo para OpenCLIP-G con mención explícita del rasgo a reemplazar ("no flame on tail, tail must not have fire"). Validado con Charmander: la llama desapareció
- **Fondo libre** — el modelo genera el fondo sin restricción. Intentos de forzar fondo blanco vía prompt fallaron consistentemente con este LoRA
- **Arquitectura 3 especialistas** — E1 (150 runs): analista Pokémon → fichas JSON con rasgos visuales. E2 (18 runs): diseñador visual por tipo → vocabulario JSON. E3 (2700 runs): redactor de prompts → 4 strings por combinación usando E1+E2

### Learnings
- El LoRA (pokesprite.safetensors) fue entrenado a 768×768 con captions BLIP en lenguaje natural — generar a 1024 produce artefactos, usar tags produce peores resultados que lenguaje natural
- negative_2 (OpenCLIP-G) con descripción explícita del rasgo a reemplazar es más efectivo que keywords cortas para romper el anclaje del LoRA en rasgos icónicos (cola de Charmander)
- guidance_scale=10 + controlnet_scale=0.4 produce llamas exageradas; guidance_scale=8.0 + controlnet_scale=0.55 da mejor balance entre transformación y legibilidad
- 5 Pokémon de prueba generados con resultados decentes: Gengar hielo y Pikachu fantasma son los más convincentes
