# file-naming
# Purpose: Enforce consistent file naming across project
# Scope: all file types (source, data, outputs, web)

All files in the project must follow consistent and descriptive naming conventions to ensure discoverability, clarity, and traceability across the codebase.

GENERAL_CONVENTIONS:
  - Use snake_case for all Python files: lowercase letters with underscores.
  - Filenames must reflect the purpose or main component of the file.
  - Avoid generic names like data.py, utils.py unless scoped in clearly named folders.

PYTHON_FILES:
  - Pipeline scripts live in pipeline/ and follow their fixed names:
      prompt_generator.py, image_generator.py, batch_runner.py
  - Global configuration: config.py (project root)
  - No test files, no debug scripts in this project.

JSON_DATA_FILES:
  - Pattern: <descriptive_name>.json
  - data/pokemons.json  – list of 150 Gen 1 Pokémon
  - data/types.json     – 18 Pokémon types with biome descriptions
  - outputs/metadata.json – auto-generated by batch_runner; not committed

OUTPUT_IMAGES:
  - Pattern: {id_pokemon}_{tipo}.png
      • id_pokemon = zero-padded number matching Pokédex ID (e.g. 001, 025, 150)
      • tipo = type name in lowercase, no spaces (e.g. water, fire, ice, ghost)
  - Examples: 001_fire.png, 025_water.png, 150_dragon.png
  - Location: outputs/images/ (gitignored)

WEB_FILES:
  - Fixed names: index.html, style.css, app.js
  - Assets in web/assets/ if needed.

CONFIGURATION:
  - config.py in the project root.
  - Constants in UPPER_SNAKE_CASE inside config.py.
